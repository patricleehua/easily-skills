## 1.通用 Skill 编写规范（推荐）

### 1.0 提供Skill加载的元数据（必须）

每一个 Skill 文件顶部必须包含 YAML 前置元数据，用于让模型准确理解：

- Skill 能做什么（Capability）
- 什么时候应该调用（Trigger Conditions）
- 什么情况下不应调用（Boundaries）

```markdown
---
name: <Skill 名称，简洁明确>
description: <触发条件 + 适用场景 + 输出类型>
license: Proprietary. LICENSE.txt has complete terms
category: <所属领域，如 workflow / coding / data>
tags: [trigger, api, debug]
---

```

#### 最重要字段：description（决定 Skill 是否会被触发）

Skill 是否能被调用，完全取决于 `description` 的语义质量。

它不是关键词匹配，而是模型判断：

> “这个 Skill 是否正好适用于用户当前请求？”

因此 description 必须清晰表达：

------

#### escription 写法公式

**动词能力 + 用户场景 + 边界约束**

------

##### 描述必须包含明确能力动词（Verbs）

模型对动作最敏感，应优先使用：

- write / generate / design
- debug / troubleshoot
- migrate / refactor
- wrap API / execute script

避免空泛词：

- “提供帮助”
- “按需加载”
- “支持工作流”

------

#####  描述必须覆盖用户真实触发意图（User Cases）

Skill 应该在用户说出类似话时触发，例如：

- “帮我写一个 Skill”
- “为什么触发失败？”
- “怎么把 API 封装成 action？”
- “生成符合某格式的输出”

建议 description 写成：

> Use this skill when the user asks to…

------

##### 必须写清楚触发边界（When NOT to use）

模型需要知道“不该调用它的时候”，否则触发会漂移。

例如：

- 不用于普通聊天
- 不回答与 Skill 无关的编程问题
- 不处理纯文案任务

------

##### description 模板示例（高触发率）

```
---
description: Use this skill when the user asks to design, implement, debug, or migrate Skills workflows,including writing Skill.md structures, wrapping APIs into actions, troubleshooting trigger failures,or converting workflows across agents.Do NOT use for general coding questions unrelated to Skills.
---
```



##### 常见触发条件应显式写进 description

------

###### 文件触发型

- 当用户上传或粘贴以下内容时调用：

| 输入类型 | 示例               |
| -------- | ------------------ |
| 字幕文件 | SRT/VTT            |
| 会议记录 | Transcript/Minutes |
| 系统日志 | Debug logs         |
| API 文档 | Swagger/OpenAPI    |

------

###### 意图触发型

- 当用户表达以下意图时调用：
- “我不会写…”
- “我需要一个方案”
- “这个为什么不触发？”
- “帮我迁移到另一个工具”

------

###### 输出格式触发型

- 当用户要求生成特定结构：
- YAML schema
- Skill.md 模板
- Workflow 执行脚本
- API action wrapper

------

### 文档级 Metadata 推荐字段（可选但强烈建议）

为了长期维护与检索，应补充：

| 字段            | 用途               |
| --------------- | ------------------ |
| version         | 管理迭代           |
| category        | 分类检索           |
| tags            | 提高语义触发召回   |
| author          | 责任归属           |
| last_update     | 避免调用过期 Skill |
| compatible_with | 指定运行平台       |

### 1.2 指令层的写法要点

建议包含以下部分：

- 角色设定（你是谁）
- 任务定义（要做什么）
- 输入假设（用户会提供什么）
- 输出结构（必须的标题、表格、字段）
- 约束与边界（禁止编造、禁止删减、必须保留原文等）
- 后续处理（若需要脚本或 MCP 工具，写清楚调用条件与步骤）

### 1.3 输出格式要稳定可复用

最好规定固定的标题、表格字段、文件命名方式，便于自动化处理与资料沉淀。

不允许输出`emoji`表情

```

  SKILL.md

  scripts/
    run.py
  references/

    rules.md
  assets/
```
